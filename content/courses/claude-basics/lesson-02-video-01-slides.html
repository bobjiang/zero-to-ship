Section 3 • 4:30
C1 — Clear

Be Specific About What You Want

Specificity does not require technical knowledge. Just describe what you want to see, what it should do, and what it should look like.

The Misunderstand Test

"Could someone misunderstand this?" If yes, add more detail.

Before vs After

Vague: "Make a website"
Clear: "Create a single-page portfolio website with a hero section that has my name and tagline, an about-me section, and a contact form at the bottom."
Section 4 • 6:00
C2 — Contextual

Give Claude the Bigger Picture

Claude performs dramatically better when it understands the "why" and "where." Think of context as answering the questions Claude would ask if it could.

▶ Project type
▶ Tech stack
▶ Who the users are
▶ What already exists

Context Is King

No context: "Write a function to process payments"
With context: "I'm building an e-commerce checkout page for a Next.js app. I'm using Stripe for payments. Write a server-side function to handle the payment process."
Section 5 • 7:30
C3 — Constrained

Set Boundaries That Guide

Constraints are guidance, not limitations. Think of them like guardrails on a bowling lane -- they keep the ball out of the gutter.

02Ship Example

"Use TypeScript and Tailwind CSS, keep components under 100 lines, and make everything mobile-responsive."

Constraint Phrases You Can Steal

"Keep it simple"
"Under 500 words"
"No external libraries"
"Make it accessible"
"Mobile-responsive"
Section 6 • 9:00
C4 — Examples

Show Claude What Good Looks Like

Examples teach Claude your style faster than any amount of explanation. You can give "good" examples and "bad" examples -- both are useful.

Show, Don't Just Tell

One example is worth a paragraph of description.

Example in Action

Vague: "Write in a friendly tone"
With example:

Good: "Hey! Let's build this together."

Bad: "Greetings, user. Construction shall proceed."

After: All 4 Cs Applied

Example "Style it like the Vercel homepage -- minimal, clean, dark mode."
