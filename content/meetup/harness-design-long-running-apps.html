The Problem

Why Naive Implementations Fall Short

Two fundamental challenges block single-agent approaches from building real applications

🧠

Context Window Degradation

Models lose coherence on lengthy tasks. Some exhibit “context anxiety” — wrapping up prematurely as they approach perceived limits.

🤖

Self-Evaluation Bias

Agents confidently praise their own work, even when quality is mediocre. Particularly acute for subjective tasks like design.

Speaker Notes

02:00 – 05:00 (3 min)
  • Context window degradation: as token count grows, the model loses track. Prior approaches hit a performance ceiling despite prompt engineering.
  • “Context anxiety” — Opus 4.5 would wrap up prematurely; Opus 4.6 largely fixed this.
  • Self-evaluation: agents rate their own output highly. No binary verification for design quality. Separate evaluator needed.
Solutions

❌ Single Agent

Long context → incoherent output. Self-review → blind to own flaws. No external feedback loop.

VS

✅ Multi-Agent Harness

Context resets + structured handoffs. Separate generator from evaluator. Concrete iteration cycles.

Speaker Notes

05:00 – 07:00 (2 min)
  • Solution to context: clear the window entirely, carry state via structured handoffs. Trade-off: orchestration complexity + token overhead.
  • Solution to self-eval: separate the generator from the evaluator. Tuning skepticism in a standalone evaluator is far more tractable.
  • This is the GAN inspiration — generator creates, evaluator judges, feedback loops back.

Frontend Design

Grading Subjective Quality

Four criteria that make design quality measurable and iteratable

🎨

Design Quality

Coherent mood/identity. Colors, typography, layout combine into a whole.

✨

Originality

Custom decisions vs. template defaults. Penalizes “AI slop” patterns.

🔧

Craft

Typography hierarchy, spacing consistency, color harmony, contrast.

📱

Functionality

Can users find actions and complete tasks without guessing?

Speaker Notes

09:00 – 12:00 (3 min)
  • Design quality + originality weighted higher — Claude already does well on craft and functionality.
  • Bias toward aesthetic risk-taking pushes outputs away from bland defaults.
  • Evaluator uses Playwright MCP: navigates the page, takes screenshots, studies implementation before scoring.
  • 5–15 iterations per generation; full runs extended to 4 hours.

Full-Stack Architecture

Three-Agent System for Application Development

📋

Planner Agent

Expands 1–4 sentence prompt into full product spec. Stays at product level — avoids granular technical detail to prevent cascading errors.

🔨

Generator Agent

Implements features from spec. React + Vite + FastAPI + SQLite. Uses git for version control. Negotiates sprint contracts with evaluator.

🔍

Evaluator Agent

Uses Playwright MCP to click-test the live app. Grades against bugs found + objective criteria. Hard thresholds trigger rework.

Speaker Notes

13:00 – 16:00 (3 min)
  • Planner is deliberately more ambitious about scope. Identifies opportunities to weave in AI features.
  • Generator works in sprints, self-evaluates before QA handoff.
  • Sprint Contract: generator and evaluator agree on “done” definition before code is written. Bridges user stories and testable implementation.
  • Evaluator grades: product depth, functionality, visual design, code quality — each with hard threshold.
Quality Assurance

Sprint Contracts & QA Tuning

Out-of-the-box, Claude is a poor QA agent. Calibrating skepticism was key.

  • ✗ Identified issues then talked itself out of them
  • ✗ Tested superficially; subtle bugs escaped
  • ✓ Multiple prompt iterations to calibrate skepticism
  • ✓ Sprint contracts define “done” before coding starts

Example QA Findings

FAIL: Rectangle fill tool

Only places tiles at drag start/end instead of filling region

FAIL: Delete entity spawn points

Handler requires both selection AND selectedEntityId, but click only sets one

FAIL: Reorder animation frames

PUT /frames/reorder defined after /{frame_id}; FastAPI matches “reorder” as integer → 422

Speaker Notes

16:00 – 18:00 (2 min)
  • Key insight: making the evaluator skeptical enough took multiple prompt iterations.
  • Sprint contracts bridge the gap between user stories and testable criteria.
  • These QA findings are real bugs found by the evaluator agent in the Retro Game Maker output.
  • Remaining limits: small layout issues, unintuitive interactions, deeply nested feature bugs.

Speaker Notes

25:00 – 26:00 (1 min)
  • Key lesson: when new models release, re-examine harness components. Strip non-load-bearing parts, add new capabilities.
  • Opus 4.5 required context resets between sessions. Opus 4.6 made that scaffolding unnecessary.
  • Every harness component encodes an assumption about what models can’t do alone.
Principles

Key Principles

1

Experiment with actual models. Read traces on realistic problems. Tune performance toward goals — don’t rely on assumptions.

2

Decompose with specialized agents. One agent per aspect of a complex task can yield substantial gains over monolithic approaches.

3

Re-examine on model upgrades. Strip non-load-bearing components. Add new capabilities. Assumptions get stale fast.

4

Better models expand the design space. They don’t eliminate interesting harness work — they shift where the interesting combinations exist.

Speaker Notes

26:00 – 28:00 (2 min)
  • These principles apply beyond this specific harness — any multi-agent system benefits from this thinking.
  • Evaluator usefulness depends on task difficulty relative to model capability. Frontier tasks benefit most.
  • Every component is an assumption about model limitations. Stress-test these regularly.
Summary

Takeaways

✓

Separate generation from evaluation. Self-evaluation bias is real. External feedback enables concrete iteration.

✓

Make subjective quality gradable. Define criteria, weight them strategically, iterate against scores.

✓

Sprint contracts bridge spec and test. Agree on “done” before code, not after.

✓

Harness components are assumptions. Re-evaluate with every model release. Today’s scaffolding may be tomorrow’s overhead.
