The Problem

Why Naive Implementations Fall Short

Two fundamental failure modes when running agents on complex, long tasks

📉

Context Window Degradation

Models lose coherence as context fills up. Some exhibit "context anxiety" — wrapping up work prematurely near perceived limits.

🤖

Self-Evaluation Bias

Agents confidently praise their own work regardless of actual quality, especially on subjective tasks without verifiable checks.

⚠

The Combined Effect

Impressive-looking interfaces with completely non-functional core mechanics. The agent thinks it's done — it's not.

Speaker Notes

03:00 – 08:00 (5 min)
  • Context window degradation: the longer the task, the worse the output. Models start cutting corners.
  • "Context anxiety" is a real observed behavior — Sonnet 4.5 would rush to finish when context got large.
  • Self-evaluation bias: if you ask the agent to judge its own work, it almost always says it's great.
  • The combined result: you get something that looks polished but doesn't actually work.

Frontend Design

Making Subjective Quality Gradable

Four criteria to evaluate design work that has no single "right answer"

  • ✓ Design Quality — Does it feel like a coherent whole?
  • ✓ Originality — Custom decisions vs. template layouts
  • ✓ Craft — Typography, spacing, color harmony
  • ✓ Functionality — Usability independent of aesthetics

Key Finding

Weighting design and originality heavily pushed models away from "AI slop" patterns.

Evaluator uses few-shot calibration to ensure consistent judgment.

Built with Claude Agent SDK + Playwright MCP for interactive evaluation.

Speaker Notes

11:00 – 16:00 (5 min)
  • How do you grade something subjective like design? You break it into concrete criteria.
  • The weight distribution matters enormously — emphasizing originality steers away from generic templates.
  • Evaluator calibration with few-shot examples prevents grading drift over multiple iterations.
  • 5-15 iteration cycles per generation, 4-hour max wall-clock time.
  • Example outcome: a museum website evolved from conventional dark theme to a spatial 3D CSS experience.

Full-Stack

Three-Agent Architecture

Scaling from frontend design to full-stack application development

📋

Planner Agent

Expands 1-4 sentence prompts into complete product specs. Identifies AI integration opportunities.

🔨

Generator Agent

Works in feature-based sprints. React + Vite + FastAPI + SQLite. Self-evaluates before handoff.

🔎

Evaluator Agent

Uses Playwright MCP for live testing. Grades against sprint contracts with hard thresholds.

Speaker Notes

17:30 – 21:00 (3.5 min)
  • The three agents form a pipeline: plan → build → evaluate → iterate.
  • Planner takes a simple prompt and creates a full product specification — this prevents scope ambiguity.
  • Generator works in sprints with version control via Git — it can roll back if needed.
  • Evaluator doesn't just check code — it actually interacts with the running application via Playwright.
  • Communication between agents uses file-based handoffs for context preservation.

Evolution

From V1 to V2

As models improve, harness design evolves — removing scaffolding that's no longer needed.

  • ✓ Removed sprint construct (Opus 4.6 handles long context)
  • ✓ Moved evaluator to single end-of-run pass
  • ✓ Improved AI agent integration within apps
  • ✓ Reduced total cost from $200 to ~$125

26:00 – 28:30 (2.5 min)
  • Each harness component encodes assumptions about model limitations. As models improve, some scaffolding becomes unnecessary.

  • This progression shows why harness design is a moving target.

Takeaways

Key Recommendations

1

Separate generation from evaluation for objective feedback on subjective tasks.

2

Use grading criteria as steering — prompting language directly shapes output character.

3

File-based handoffs for reliable multi-agent communication and context preservation.

4

Negotiate contracts before building to bridge specs and implementation details.

5

Re-examine harnesses per model release — remove non-load-bearing complexity.

6

Accept evaluator limitations — no approach captures all edge cases and subtle bugs.
